# Findings: ONNX v2 Web Inference Noise Bug

## Confirmed Root Cause

**The ONNX encoder (`mimi_encoder.onnx`) applies normalization before projection, but the PyTorch reference does NOT. This causes the voice conditioning vector to be numerically wrong, which makes the model generate noise instead of speech.**

---

## 1. The Two-Phase Voice Processing Pipeline

Understanding the EXACT data flow is critical.

### PyTorch Reference (`tts_model.py`)

**Phase A — Voice Conditioning (encoder side):**
```python
# tts_model.py line 373-381: _encode_audio()
encoded = self.mimi.encode_to_latent(audio)     # [B, 32, T']
latents = encoded.transpose(-1, -2)              # [B, T', 32]
conditioning = F.linear(latents, self.flow_lm.speaker_proj_weight)  # [B, T', 1024]
# ← NO normalization! Raw 32-dim latents are projected directly to 1024-dim
```

**Phase B — Latent Decoding (decoder side):**
```python
# tts_model.py line 442-444: _decode_audio_worker()
mimi_decoding_input = latent * self.flow_lm.emb_std + self.flow_lm.emb_mean  # un-normalize
transposed = mimi_decoding_input.transpose(-1, -2)
quantized = self.mimi.quantizer(transposed)
```

So `emb_std`/`emb_mean` are **only used on the decoder side** to un-normalize the latents generated by the flow LM before passing them to Mimi.

---

## 2. What the ONNX Wrapper Does (The Bug)

**`MimiEncoderWrapper.forward()` (wrappers.py):**
```python
latents = encoded.transpose(-1, -2)              # [B, T', 32]
latents = (latents - self.emb_mean) / self.emb_std   # ← WRONG: normalizes before projection
latents = F.linear(latents, self.speaker_proj_weight) # [B, T', 1024]
```

**This is wrong.** The PyTorch reference does `F.linear(raw_latents, weight)` — no normalization before projection.

The `emb_mean`/`emb_std` normalization before encoder projection was incorrectly added during the refactor and was not present in the original PyTorch model. The parity test passes because it compares the ONNX output against the *wrapper's* output (which also normalizes), not against the true *PyTorch model* output.

---

## 3. Verification: Why Parity Test Passed But Audio Is Wrong

The `compare_onnx_pytorch.py` script calls:
```python
pt_encoder = MimiEncoderWrapper(tts_model.mimi, tts_model.flow_lm.speaker_proj_weight,
                                 emb_std=tts_model.flow_lm.emb_std,
                                 emb_mean=tts_model.flow_lm.emb_mean)
pt_out = pt_encoder(test_audio)
```

So the reference output IS the wrapper (already applying normalization). The test proves the ONNX and the wrapper agree — but **both are wrong relative to the real model behavior**.

---

## 4. Secondary Issue: BOS Concatenation Order

Looking at the PyTorch reference (tts_model.py line 884-885):
```python
# PyTorch: BOS is prepended BEFORE the voice conditioning
prompt = torch.cat([self.flow_lm.bos_before_voice, prompt], dim=1)
# Result: [BOS, voice_frames]  → fed to flow_lm
```

Looking at the ONNX wrapper (`export_flow_lm.py` line 181):
```python
# ONNX wrapper: BOS is placed BETWEEN text and voice
combined = torch.cat([text_embeddings, bos_to_use, input_], dim=1)
# During voice conditioning: text_embeddings=voice_emb, input_=empty_seq
# Result: [voice_frames, BOS, empty]  → different order!
```

This is a **reversed order**. The PyTorch model first processes `[BOS, voice_frames]`, but the ONNX model processes `[voice_frames, BOS]`. This means the BOS token that should prime the voice context arrives **after** the voice information instead of before it.

---

## 5. Summary of Bugs

| # | Bug | Location | Effect |
|---|-----|----------|--------|
| **1** | `MimiEncoderWrapper` incorrectly normalizes before projection | `wrappers.py` lines 191-195 | Voice embedding is computed from wrong values → fundamentally wrong conditioning → **noise** |
| **2** | BOS is appended AFTER voice instead of BEFORE | `export_flow_lm.py` line 181 | Causal model sees voice before its BOS primer → different attention patterns |
| **3** | Parity test compares wrapper vs wrapper | `compare_onnx_pytorch.py` | Masks bug #1 — test passes even though results are wrong |

## 6. Required Fixes

### Fix 1: Remove normalization from MimiEncoderWrapper

```python
# wrappers.py - MimiEncoderWrapper.forward()
def forward(self, audio):
    encoded = self.mimi.encode_to_latent(audio)
    latents = encoded.transpose(-1, -2)          # [B, T', 32]
    
    # NO normalization here! Apply projection directly.
    if self.speaker_proj_weight is not None:
        latents = torch.nn.functional.linear(latents, self.speaker_proj_weight)
    return latents
    # emb_std, emb_mean are NOT needed in the encoder wrapper at all
```

### Fix 2: Fix BOS concatenation order

The PyTorch model does:  
`prompt = cat([bos_before_voice, voice_conditioning], dim=1)` → then feeds `prompt` as `text_embeddings`

So in the web worker, the voice embedding passed to `flow_lm_main.onnx` should already have BOS prepended before it, NOT appended inside the ONNX graph after it.

**Option A**: Keep BOS auto-insertion in the ONNX graph, but:
- The ONNX graph must accept `text_embeddings = pure_voice_emb` (no BOS)  
- The graph internally does `cat([BOS, text_embeddings, input_], dim=1)`

**Option B**: Remove BOS from the ONNX graph and let the web worker prepend it explicitly.

Currently the ONNX graph prepends in this order: `[text_embeddings, BOS, input_]` which is `[voice, BOS, seq]`. The PyTorch model uses `[BOS, voice, seq]`. The order matters for the causal attention mask.

### Fix 3: Fix the parity test

```python
# compare_onnx_pytorch.py - use REAL PyTorch model output, not the wrapper
def compare_encoder(ort_session, tts_model):
    test_audio = torch.randn(1, 1, 24000)
    
    # Reference: what PyTorch actually does
    with torch.no_grad():
        encoded = tts_model.mimi.encode_to_latent(test_audio)
        latents = encoded.transpose(-1, -2)
        pt_out = F.linear(latents, tts_model.flow_lm.speaker_proj_weight).numpy()
    
    onnx_out = ort_session.run(None, {"audio": test_audio.numpy()})[0]
    assert_allclose("Mimi Encoder", pt_out, onnx_out)
```

## 7. Action Plan

1. **Fix `MimiEncoderWrapper`**: Remove the `(latents - emb_mean) / emb_std` step entirely. The projection `F.linear(latents, speaker_proj_weight)` takes raw 32-dim latents.

2. **Fix BOS order in `FlowLMMainWrapper`**: Change the concatenation from `[text_emb, BOS, input_]` to `[BOS, text_emb, input_]` (to match PyTorch's `[bos_before_voice, prompt]` order before the backbone runs).

3. **Re-export `mimi_encoder.onnx` and `flow_lm_main.onnx`** for english_v2.

4. **Fix parity test** to compare against *raw PyTorch model* flow, not the wrapper.

5. **Re-export all other languages** and verify.
